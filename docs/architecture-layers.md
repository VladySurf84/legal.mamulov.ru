# Архитектура слоев учета

В проекте мы разделяем четыре разные вещи:

- как мы получили данные;
- что пришло из источников;
- что мы считаем каноническим бизнес-документом;
- какие выводы мы из этого считаем.

## Общая схема

```text
Acquisition Layer
  запуски синхронизаций, API-запросы, файлы, ручной ввод, расписание

        -> сохранение полученных данных

Source Layer
  сырые записи из банков, файлов, API, ЭДО, отчетов бухгалтера

        -> нормализация / распознавание / сопоставление

Canonical Layer
  наша версия бизнес-документов и их сторон

        -> интерпретация / пересчет

Interpretation Layers
  деньги, НДС, обязательства, связи отчетов с документами
```

## Acquisition Layer

Acquisition layer отвечает за технический процесс получения данных.

Этот слой не говорит, что означает документ. Он отвечает на вопросы:

- каким способом мы получили данные;
- какой API-ключ или аккаунт использовался;
- какой endpoint был вызван;
- какой запрос ушел;
- какой ответ пришел;
- какой файл был загружен;
- кто и когда запустил импорт;
- была ли ошибка;
- какой период или курсор синхронизации использовался.

### `legal.api_credentials`

Ключи и учетные данные для внешних API.

Примеры:

- токены Тинькофф Бизнес;
- будущие ключи ЭДО;
- будущие ключи других банков или сервисов.

Важно: это не только банковские ключи. Таблица предназначена для всех внешних API-аккаунтов.

### `legal.api_sync_runs`

Запуски синхронизаций.

Пример:

```text
запустили синхронизацию Тинькофф за период 2026-05-01..2026-06-20
```

Эта таблица отвечает за общий процесс: когда стартовал запуск, чем закончился, сколько записей обработано, какая была ошибка.

### `legal.api_sync_requests`

Конкретные HTTP-запросы внутри запуска синхронизации.

Пример:

```text
GET /api/v1/bank-statement
account = 408...
status = 200
response_body = оригинальный текст ответа
response_json = распарсенный JSON
```

Эта таблица нужна для аудита: можно вернуться к оригинальному ответу API и понять, откуда взялась каждая запись.

### `legal.vat_book_imports`

Сейчас эта таблица частично играет роль acquisition layer для файлов книг покупок и продаж.

Она хранит:

- какое юрлицо;
- какой год и квартал;
- книга покупок или продаж;
- имя исходного файла;
- путь сохраненного файла;
- хеш файла;
- размер файла;
- количество строк.

Позже эту роль можно будет обобщить через `source_records` и `source_record_files`, но текущая таблица остается рабочей.

### `legal.import_runs`

Запуски ручных или файловых импортов.

Пример:

```text
загрузили файл выписки Ozon Банка в формате 1CClientBankExchange
```

Эта таблица отвечает за общий процесс файлового импорта: когда стартовал запуск, каким статусом закончился, сколько строк и операций обработано, какая ошибка произошла.

### `legal.uploaded_files`

Архив загруженных файлов.

Пример:

```text
оригинальный файл выписки Ozon Банка сохранен в storage/app/private/legal/imports/ozon-bank/...
```

Важные поля:

- `import_run_id`;
- `provider`;
- `file_role`;
- `source_file_name`;
- `stored_path`;
- `mime_type`;
- `file_sha256`;
- `file_size`;
- `uploaded_by_user_id`;
- `uploaded_at`.

### `legal.kassa` и `legal.kassa_article`

Ручной ввод денежных операций. Таблица `legal.kassa` остается acquisition/source-таблицей, потому что данные в нее могут вноситься вручную. После миграции она связана с каноническим слоем:

- `legal.kassa.legal_id` хранит ИНН нашего юрлица;
- `legal.kassa.document_id` указывает на созданный канонический документ;
- тип канонического документа - `manual_cash_operation` / `Ручная денежная операция`;
- триггер `legal.sync_kassa_document()` создает или обновляет `legal.documents`, `legal.source_records`, `legal.document_sources` и `legal.document_parties`;
- `legal.kassa_article` используется как статья операции и как имя второй стороны, если явный контрагент не задан.

По умолчанию старое поле `user_id` заменено на `legal_id = 504025873801`.

### Будущие таблицы Acquisition Layer

Возможные будущие таблицы:

- `legal.scheduler_runs` - фактические запуски планировщика;
- `legal.scheduler_jobs` - задания внутри планировщика.

Пример цепочки для банка:

```text
api_sync_runs
  "запустили синхронизацию Тинькофф"

api_sync_requests
  "получили ответ /api/v1/bank-statement"

source_records
  "из ответа извлечена строка банковской выписки"

source_record_bank_details / source_record_parties / source_record_amounts
  "сохранили банковские детали, стороны и суммы как они пришли из источника"

documents
  "мы считаем, что это банковская операция / платежка"

document_sources
  "связали канонический документ с сырой строкой выписки"

money_edges
  "из этого следует движение денег"
```

Пример цепочки для файла Ozon Банка:

```text
import_runs
  "запустили импорт файла Ozon"

uploaded_files
  "сохранили оригинальный файл и его sha256"

documents / document_bank_transaction
  "текущий код сразу создает банковские документы"

money_edges
  "из этого следует движение денег"
```

Позже между `uploaded_files` и `documents` должен появиться явный `source_records`.

## Source Layer

Source layer хранит то, что пришло из внешнего или ручного источника, максимально близко к оригиналу. Эти таблицы не должны пытаться выразить окончательный бизнес-смысл. Это доказательства и первичные наблюдения.

### `legal.source_records`

Общая шапка любого сырого факта из источника.

Примеры:

- строка банковской выписки Тинькофф;
- строка файла Озон Банка;
- XML УПД, загруженный вручную;
- XML УПД, полученный из ЭДО Лайт;
- строка книги покупок или продаж, импортированная из XML бухгалтера.

Важные поля:

- `source_system` - откуда пришло, например `tinkoff`, `ozon_file`, `edo_light`, `manual_upload`, `accountant_vat_book`;
- `source_channel` - более узкий канал, например `api`, `xml_upload`, `file_upload`;
- `source_record_type` - тип сырого факта, например `bank_statement_line`, `upd_xml`, `vat_book_row`;
- `external_id` - внешний идентификатор в источнике, если он есть;
- `external_hash` - хеш для идентификации, если стабильного внешнего id нет;
- `source_api_sync_request_id` - API-запрос, который принес запись;
- `source_vat_book_entry_id` - существующая строка книги НДС, пока мы переводим проект на source layer;
- `source_file_path` - путь к исходному файлу;
- `raw_payload` - исходные данные в распарсенном виде;
- `metadata` - служебные данные импорта.

Уникальность защищается индексами:

- `source_system`, `source_record_type`, `external_id`;
- `source_system`, `source_record_type`, `external_hash`.

### `legal.source_record_parties`

Стороны ровно в том виде, как они были указаны в источнике.

Примеры:

- плательщик и получатель из банковской выписки;
- продавец и покупатель из XML УПД;
- контрагент из строки книги покупок.

Это не финальные нормализованные стороны документа. Это формулировки и идентификаторы из источника.

Важные поля:

- `source_record_id`;
- `source_party_role`;
- `role_index`;
- `party_name`;
- `inn`;
- `kpp`;
- `account_number`;
- `bank_bic`;
- `bank_name`;
- `address`;
- `raw_payload`.

### `legal.source_record_amounts`

Суммы ровно в том виде, как они были указаны в источнике.

Примеры:

- сумма с НДС;
- сумма без НДС;
- сумма НДС;
- сумма банковской операции со знаком.

Важные поля:

- `source_record_id`;
- `amount_type`;
- `amount`;
- `currency`;
- `tax_rate`;
- `raw_payload`.

### `legal.source_record_files`

Файлы, связанные с сырой записью.

Примеры:

- оригинальный XML УПД;
- ZIP-архив из ЭДО;
- загруженный PDF;
- оригинальный XML книги покупок или продаж.

Важные поля:

- `source_record_id`;
- `file_role`;
- `source_file_name`;
- `stored_path`;
- `mime_type`;
- `file_sha256`;
- `file_size`;
- `encoding`.

### `legal.source_record_product_marks`

Коды маркировки товаров, найденные в сырой записи.

Это новый универсальный аналог старой идеи `ch_edo_doc_cise` из `api.mamulov.ru`.

Примеры:

- КИЗ;
- СИЗ;
- DataMatrix;
- CIS-коды из XML УПД.

Важные поля:

- `source_record_id`;
- `mark_code`;
- `mark_type`;
- `product_group`;
- `gtin`;
- `serial_number`;
- `raw_payload`.

### `legal.source_record_bank_details`

Банковские детали для сырой записи, которая является строкой банковской выписки или похожим банковским наблюдением.

Важно: это не значит, что банковская операция не является документом. Это значит, что строка банковской выписки имеет особую форму данных.

Важные поля:

- `source_record_id`;
- `bank_account_id`;
- `bank_id`;
- `account_number`;
- `external_operation_id`;

- `draw_date`;

### `legal.source_record_document_details`

Детали сырого документоподобного источника.

Таблица намеренно не называется `edo_documents`, потому что один и тот же УПД может прийти из ЭДО Лайт, Диадока, ручной загрузки XML, PDF или другого канала.

Важные поля:

- `source_document_type`;
- `source_direction`;
- `source_status`;
- `document_number`;
- `document_date`;

- `external_status`;
- `processed_at`.

### `legal.source_record_vat_book_details`

Детали сырой записи, созданной из строки книги покупок или продаж бухгалтера.

Важные поля:

- `source_record_id`;
- `vat_book_import_id`;
- `vat_book_entry_id`;
- `legal_id`;

## Уже существующие source-таблицы

Эти таблицы появились до общего source layer и пока используются текущими экранами и пересчетами.

### `legal.document_bank_transaction`

Текущая таблица банковских операций в новой структуре.

Исторически она одновременно играла две роли:

- банковская детализация документа;
- сырое банковское наблюдение.

Для Тинькофф API новый импорт уже пишет сырой источник в `source_records`, а связь с каноническим документом - в `document_sources`. Сама `document_bank_transaction` пока остается рабочей detail-таблицей для текущих экранов и расчетных слоев.

### `legal.vat_book_imports`

Шапка импорта XML-файла книги покупок или продаж бухгалтера.

### `legal.vat_book_entries`

Строки книги покупок и продаж бухгалтера.

Это отчетные данные бухгалтера, а не гарантированно первичные документы. Одна строка может соответствовать УПД, счету-фактуре или другому документу, но сама строка книги является источником для сравнения.

## Canonical Layer

Canonical layer хранит нашу текущую лучшую версию бизнес-документов.

### `legal.documents`

Главная таблица канонических документов.

Примеры типов документов:

- банковская операция / платежное поручение;
- УПД;
- акт;
- счет;
- акт сверки / начальное сальдо.

Важные поля:

- `document_id`;
- `document_type_id`;
- `document_date`;
- `document_number`;
- `title`;
- `amount`;
- `currency`;
- `status`;
- `source_system`;
- `external_id`;
- `external_hash`;
- `metadata`.

Поля `source_system`, `external_id`, `external_hash` остаются для совместимости. Долгосрочно предпочтительная связь с источниками - через `legal.document_sources`.

### `legal.document_types`

Справочник канонических типов документов.

### `legal.document_parties`

Канонические стороны документа. Это уже наша интерпретация сторон, а не обязательно точная копия из источника.

Примеры ролей:

- плательщик;
- получатель;
- продавец;
- покупатель;
- грузоотправитель;
- грузополучатель;
- подписант.

### `legal.document_party_roles`

Справочник канонических ролей сторон документа.

### `legal.document_sources`

Мост между каноническими документами и сырыми источниками.

Эта таблица позволяет нескольким источникам подтверждать один и тот же бизнес-документ.

Пример:

```text
documents:
  document_id = 100
  type = upd

document_sources:
  document_id = 100, source_record = входящий XML из ЭДО Лайт у Мамулова
  document_id = 100, source_record = исходящий XML из ЭДО Лайт у Бертос
  document_id = 100, source_record = строка книги покупок бухгалтера
```

Важные поля:

- `document_id`;
- `source_record_id`;
- `source_item_key` - часть источника, если один источник порождает несколько документов;
- `source_role` - например `evidence`, `primary`, `counterparty_copy`;
- `confidence`;
- `matched_by`;
- `matched_at`;
- `metadata`.

## Interpretation Layers

Interpretation layers - это расчетные слои. Их должно быть можно очистить и пересчитать заново при изменении алгоритма.

### `legal.money_edges`

Денежный слой.

Сейчас строится сервисом `App\Services\Layers\MoneyLayerBuilder` из:

- банковских документов через `legal.document_bank_transaction`;
- ручных денежных операций `manual_cash_operation`, которые приходят из `legal.kassa`.

Смысл:

- кто платит;
- кто получает;
- сумма;
- дата;
- документ-источник.

Это не сырая банковская транзакция, а наша интерпретация движения денег.

### `legal.vat_events`

Слой НДС.

Сейчас строится из:

- строк книг покупок и продаж бухгалтерии через `App\Services\Layers\VatLayerBuilder`;
- банковских назначений платежей через `App\Services\Layers\BankVatLayerBuilder`.

Смысл:

- входящий НДС;
- исходящий НДС;
- источник НДС;
- период;
- контрагент.

### `legal.accountant_report_links`

Связи строк книги покупок/продаж бухгалтера с банковскими транзакциями.

Используется, чтобы отвечать на вопросы:

- какие оплаты закрываются какими строками книги покупок;
- по каким оплатам бухгалтерский документ еще не найден;
- какие строки книги продаж связаны с банковскими приходами.

Это расчетная таблица результата сопоставления, а не первичный источник истины.

### Будущий слой: `legal.obligation_edges`

Планируемый слой обязательств.

Примеры:

- УПД создает кредиторскую или дебиторскую задолженность;
- платеж закрывает задолженность;
- акт или корректировка меняет задолженность.

## Практическое правило

Когда добавляем новую таблицу, проверяем:

- если таблица хранит то, что сказал внешний источник, файл или API, это source layer;
- если таблица хранит процесс получения данных, запуск, запрос, ответ или загруженный файл, это acquisition layer;
- если таблица хранит то, что мы считаем бизнес-документом, это canonical layer;
- если таблицу можно удалить и пересчитать из других таблиц, это interpretation layer;
- если значение важно для аудита, сначала сохраняем оригинальное значение источника, а потом нормализуем;
- если два источника описывают один бизнес-документ, связываем их через `document_sources`, а не заставляем один источник перезаписывать другой.
